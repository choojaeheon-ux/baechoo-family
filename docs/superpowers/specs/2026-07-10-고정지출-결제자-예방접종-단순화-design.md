# 고정지출 결제자 표시 · 예방접종 단순화

작성일: 2026-07-10

두 가지 독립 변경을 한 스펙으로 묶는다. 공통점은 "상태는 저장되는데 화면에 안 보여서 사용자가 기능이 죽었다고 판단한다"는 점이다.

거래내역 결제수단별 모아보기는 이번 범위에서 **제외**한다(사용자 지시).

## 1. 고정지출 체크리스트 — 결제자 표시·변경

### 문제

`components/budget/CalendarView.tsx`의 `member` 상태(추추/배찌)는 `pay()`가 만드는 거래의 결제자로만 쓰인다. 체크리스트 행 어디에도 결제자가 표시되지 않으므로, 세그먼트 버튼을 눌러 색이 바뀌어도 화면상 달라지는 것이 없다. 이미 체크한 항목의 결제자는 체크 해제 후 재체크해야만 바뀐다.

버튼은 정상 동작한다(Playwright로 `setMember` → className 전환 확인). 결과가 보이지 않는 것이 결함이다.

부수 결함: 세그먼트 버튼 높이가 30px로 권장 탭 타깃(44px)에 못 미치고, 스크롤 위치에 따라 하단 고정 내비게이션(`z-40`)에 덮여 탭이 닿지 않는 구간이 있다.

### 변경

**세그먼트에 `결제자` 라벨을 붙인다.** 이름 두 개만 떠 있으면 필터로 오해된다. 버튼은 `min-h-11`(44px)로 키운다.

**체크된 행의 우측 알약 자리에 결제자 칩을 띄운다.** 미체크 행은 지금처럼 D-day 알약을 유지한다.

```
결제자  [추추] [배찌]
─────────────────────────────
☑ 🏠 월세          ₩700,000  [추추]   ← 칩 탭 = 결제자만 토글
☐ 🎬 애플뮤직        ₩8,900    D-5     ← 미체크는 D-day 그대로
```

**결제자 칩을 누르면 결제자만 바뀐다.** 체크 상태는 유지된다. 멤버가 둘뿐이므로 토글로 충분하다.

```ts
async function togglePayer(d: DueItem) {
  if (!d.paidTxn) return;
  const next: Member = d.paidTxn.member === "chuchu" ? "baejji" : "chuchu";
  await saveTransaction({ ...d.paidTxn, member: next });
}
```

`repo.saveTransaction`은 `id` 기준 upsert이므로(`lib/repo.ts:642`) 기존 거래의 `member`만 갈아끼우면 된다.

### 구조

현재 행 전체가 하나의 `<button>`이다. 그 안에 칩 버튼을 넣으면 버튼이 중첩되어 HTML이 깨진다. 행을 `<div>`로 바꾸고 내부를 두 형제로 나눈다.

- `<button>`: 체크 동그라미 + 아이콘 + 이름/출금일 → `paid ? unpay(d) : pay(d)`
- 우측 컬럼: 금액(항상) + `paid ? <결제자 칩 버튼> : <D-day Pill>`

`pay()`, `unpay()`, 할부 카운터 로직은 건드리지 않는다.

### 검증

컴포넌트 테스트 인프라가 없으므로(현재 테스트는 `lib/compute.test.ts`, `lib/pnl.test.ts` 두 개뿐) 로컬 dev 서버 + Playwright로 실제 흐름을 구동해 확인한다.

- 배찌 선택 → 미체크 항목 체크 → 행에 `배찌` 칩이 뜬다
- 칩 탭 → `추추`로 바뀌고 체크는 유지된다
- 행 탭 → 체크 해제되고 D-day 알약으로 돌아온다
- 세그먼트 버튼 실측 높이 ≥ 44px

## 2. 예방접종 — 차수 제거, 연 1회 체크

### 문제

`BaechooVaccine`이 차수별 접종 기록(`doses: {n, date}[]`)과 주기(`intervalMonths`), 다음 예정일(`nextDue`)을 모두 들고 있다. 배추의 정기 백신 5종은 전부 연 1회라 차수 개념이 불필요하고, 상세 시트의 차수 추가/삭제 UI 때문에 체크가 번거롭다.

### 데이터 모델

`doses`·`intervalMonths`·`nextDue`를 버리고 최근 접종일 하나만 남긴다.

```ts
export interface BaechooVaccine {
  id: string;
  name: string;
  lastDone: string | null;  // 최근 접종일 (YYYY-MM-DD), null = 미접종
  memo: string | null;
  createdAt: string;
}
```

`VaccineDose` 타입은 삭제한다.

체크 여부와 다음 예정일은 저장하지 않고 `lastDone`에서 파생한다. 그래서 1년이 지나는 순간 체크가 저절로 풀리며 다시 리마인드가 뜬다 — 이것이 "일 년에 한 번 체크"의 구현이다.

접종일 하나는 반드시 남는다. 이게 없으면 리마인드를 언제 다시 띄울지 계산할 수 없다. 버리는 것은 차수 번호와 과거 이력이다.

### 순수 함수 (`lib/vaccine.ts` 재작성)

```ts
export const VACCINE_INTERVAL_MONTHS = 12;

// 다음 접종 예정일 = 최근 접종일 + 12개월. 미접종이면 null.
export function vaccineNextDue(lastDone: string | null): string | null;

// 올해 접종 완료 여부 = 다음 예정일이 아직 오지 않음.
export function vaccineDone(lastDone: string | null, today: string): boolean;
```

경계 규칙:

- `lastDone === null` → `nextDue = null`, `done = false` (미접종)
- `nextDue > today` → `done = true`
- `nextDue === today` → `done = false` (오늘이 접종일 = 맞아야 함)
- `nextDue < today` → `done = false` (기한 지남)

`latestDose`, `nextDoseNumber`는 삭제한다.

TDD로 작성한다. `lib/vaccine.test.ts` 신규 — 위 네 가지 경계와 윤년(`2024-02-29` + 12개월)을 덮는다.

### 화면 (`VaccineList.tsx`)

체크박스를 토글로 만든다. 체크 → `lastDone = 오늘`, 해제 → `lastDone = null`.

| 상태 | 체크박스 | 부제 | 우측 알약 |
|---|---|---|---|
| 미접종 | 빈칸 | `접종 기록 없음` | `접종 필요` (coral) |
| 기한 지남 | 빈칸 | `2025-06-01 접종` | `접종 필요` (coral) |
| 접종 완료 | 채움 | `2026-07-10 접종 · 다음 2027-07-10` | `D-365` (30일 이내면 gold, 아니면 stone) |

정렬은 리마인드가 목적이므로 미완료를 위로 올린다. 순서는 세 덩이다.

1. **기한 지남** — 오래 밀린 것부터 (`nextDue` 오름차순)
2. **미접종** — 이름순
3. **접종 완료** — 다음 예정일 가까운 순 (`nextDue` 오름차순)

미접종을 기한 지남보다 뒤에 두는 이유는, 접종한 적 없는 백신은 배추에게 애초에 필요 없는 항목일 수 있어서다. 실제로 밀린 것이 먼저 눈에 들어와야 한다.

`STANDARD_VACCINES` 5종 일괄 추가는 그대로 두되, 새 필드 형태로 만든다.

### 상세 시트 (`forms.tsx` `VaccineDetailSheet`)

차수 추가/삭제 UI, 주기(개월) 입력, 다음 예정일 입력을 모두 걷어낸다. 남기는 것: **이름 / 접종일 / 메모 / 삭제**.

### 저장 계층

`lib/repo.ts`

- `toVaccine` / `fromVaccine` → `last_done` 한 컬럼만 매핑
- 로컬스토리지 경로에 `normalizeVaccine`을 추가한다. 구버전 레코드는 `doses`를 갖고 `lastDone`이 없으므로, `doses`의 최댓값 날짜를 `lastDone`으로 끌어올린다. `normalizeExam`과 같은 패턴이다.

### 마이그레이션 `supabase/migrations/0017_vaccine_simplify.sql`

`0013`이 만든 `last_done`을 `0014`가 지웠던 자리라, 되살리는 셈이다.

```sql
alter table baechoo_vaccines add column if not exists last_done date;

update baechoo_vaccines
set last_done = (
  select max((d->>'date')::date)
  from jsonb_array_elements(doses) d
)
where doses is not null
  and jsonb_array_length(doses) > 0
  and last_done is null;

alter table baechoo_vaccines drop column if exists doses;
alter table baechoo_vaccines drop column if exists next_due;
alter table baechoo_vaccines drop column if exists interval_months;

drop index if exists idx_baechoo_vaccines_next_due;
create index if not exists idx_baechoo_vaccines_last_done
  on baechoo_vaccines(last_done);
```

기존 `doses`의 최근 날짜를 `last_done`으로 이관하므로 리마인드가 끊기지 않는다.

`drop column`은 되돌릴 수 없다. 적용 전 `baechoo_vaccines`의 현재 내용을 조회해 로그로 남긴다.

### 검증

- `npm test` — `lib/vaccine.test.ts` 신규 통과, 기존 테스트 회귀 없음
- `npx tsc --noEmit` — `doses`·`intervalMonths` 잔존 참조 0
- 마이그레이션 적용 후 Playwright로 배추 탭 구동: 체크 → 부제·알약 전환 → 해제 → 원복
