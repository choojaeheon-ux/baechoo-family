# 배추가족 가족 P&L — 설계 명세

- 작성일: 2026-07-03
- 상태: 설계 확정 (구현 착수 전)
- 접근: C안 2단계 — ① 회계모델을 문서로 합의(본 문서) → ② PWA 손익탭으로 이식

## 1. 목적

가족 재무를 **경영자 관점의 손익계산서(P&L)**로 조망한다. 기존 가계부가 일·월 단위 거래를 굴리는 도구라면, 가족 P&L은 그 위에 얹는 **손익 레이어**다.

핵심 질문에 답한다:
- 연 단위 예산 농사를 어떻게 짜는가
- 매달 BEP를 달성하는가
- 저축·투자는 계획대로 진행되는가
- 우리 가족의 운영이익률은 월/연 단위로 몇 %인가

## 2. 회계모델 (하이브리드)

기업 손익계산서의 **용어**(매출·매출원가·운영이익·운영이익률)를 쓰되, 원가/판관비의 **경계는 가정 지출의 고정비/변동비**로 정의한다.

```
매출(총수입)                    급여·기타수입
 − 매출원가(고정비)             주거·공과·보험·구독·할부·청약 등 안 줄어드는 필수·부채상환
 − 선저축·적립투자              적금·Goal 적립 — 미래 몫을 원가처럼 먼저 뗀다
 = 가처분이익                   변동비에 쓸 수 있는 재원
 − 판관비(변동비)               식비·외식·쇼핑·여가·병원·육아·배추 등 줄일 수 있는 생활비
 = 운영이익(잉여)               저축까지 하고도 남은 진짜 여윳돈
```

### 지표
- **총이익률** = 가처분이익 ÷ 매출
- **운영이익률** = 운영이익 ÷ 매출
- **BEP(손익분기)** = 운영이익 0 지점. 즉 "고정비 + 계획저축 + 변동비를 수입으로 딱 감당하는 상태". 운영이익 > 0 이면 **계획한 저축을 다 하고도 흑자**.

### 시간축
- **월 뷰**: 해당 월의 P&L, BEP 달성 여부
- **연 뷰**: 12개월 누적 + 월별 추이

### 설계 결정 근거
- **저축을 원가처럼 선취**: Pay-yourself-first. 저축을 "남으면 하는 것"이 아니라 필수 비용으로 먼저 확보. 운영이익이 곧 "계획 저축 이후의 진짜 잉여"가 되어 재무 건전성을 보수적으로 본다.
- **하이브리드 용어**: 재헌님이 매출원가·운영이익 등 경영 용어로 재무를 조망하고 싶어함. 다만 가정엔 순수 매출원가 대응이 없으므로 고정/변동 경계로 재정의.

## 3. 카테고리 → 손익항목 매핑 규칙

현재 가계부 데이터 구조(`lib/types.ts`)를 그대로 활용해 자동 분류한다.

| 손익 항목 | 매핑 대상 | 판정 근거 |
|---|---|---|
| **매출(수입)** | 급여, 기타수입 | `Category.type === "income"` |
| **매출원가(고정비)** | `RecurringExpense`(kind: fixed·subscription·installment), 청약, 할부 | 고정지출로 등록됨 / 부채상환·주택청약 |
| **선저축·적립투자** | 적금 카테고리, `Goal` 적립분 | 저축성 항목 |
| **판관비(변동비)** | 식비·생활용품·육아·병원·취미·배추·기타 | 나머지 지출. `habitTag`(배달·커피·외식·쇼핑·택시·구독)로 "줄일 수 있는 변동비" 세분 |
| **손익 제외** | 카드값 | 결제수단 정산 — 각 카테고리 지출과 이중계상되므로 제외 |

### 매핑 원칙
- 지출의 고정/변동 판정은 **1순위 `RecurringExpense` 등록 여부**, 2순위 카테고리, 3순위 `habitTag`.
- 매핑은 카테고리마다 손익분류 속성을 다는 방식으로 명시화한다(2단계 PWA에서 데이터 모델로 구현). 자동 판정을 기본값으로 하되 수동 오버라이드를 허용.

### 미확인 사항 (재헌님 실제 운용 확인 필요)
- **카드값 처리**: 카드값을 별도 카테고리로 기록하는지, 아니면 각 지출을 카테고리별로 직접 다는지에 따라 이중계상 방지 로직이 달라진다. 현 설계는 "카드값=정산이므로 손익 제외"를 기본 가정.
- **적금·청약의 성격**: 청약을 고정비(주거 준비)로 볼지 선저축으로 볼지. 현 설계는 청약=고정비, 적금=선저축.

## 4. 자산 스냅샷 대조 (P&L ↔ 실제 자산)

손익(기간 흐름)과 잔액(시점 자산)은 다른 표다. 이 모델은 **P&L을 중심으로 하되 월말 자산 스냅샷을 경량 연결**한다.

- 매월 말 **총자산**(현금·계좌·투자 잔액 합계)을 수동 입력
- **운영이익(잉여) ≈ 월말 총자산 − 월초 총자산** 을 대조
- 차이가 크면 거래 누락·미기록을 점검하는 신호로 사용

```
[P&L]  운영이익(잉여)      +500,000
                ↓ 대조
[자산] 월초 총자산       50,000,000
       월말 총자산       50,480,000
       실제 증가           +480,000
       차이 20,000 → 누락 점검
```

자산−부채=순자산 추이, 부채·자산 상세 관리는 **범위 밖**(2단계 이후 별도 확장).

## 5. 산출물

### 1단계 (현재 — 문서)
1. **회계모델 규칙서** (`.md`): 본 문서의 2~4장을 완성된 규칙 문서로 정리. 손익구조·매핑표·공식·BEP 정의·자산대조 규칙.
2. **2026 연간예산 P&L 템플릿**: 1~12월 × (매출 / 고정비 / 선저축 / 변동비 / 운영이익 / 운영이익률 / BEP달성) 빈 표. 재헌님이 목표 숫자를 채운다. 이 템플릿이 2단계 PWA 화면의 뼈대가 된다.

### 2단계 (다음 — PWA)
- 가계부 탭에 **"P&L / 손익" 서브탭** 추가
- `transactions` · `budgets` · `recurring` · `goals`를 손익 구조로 집계하는 **라이브 손익계산서**
- 월/연 토글, BEP 게이지, 운영이익률, 계획 대비 실적, 자산 스냅샷 입력·대조
- 규칙서가 곧 이 화면의 설계도

## 6. 범위 (YAGNI)

**포함**: 하이브리드 P&L 회계모델 정의, 카테고리 매핑 규칙, 월/연 뷰, BEP, 자산 스냅샷 경량 대조, 1단계 규칙서+템플릿.

**제외(현 단계)**: 순자산(자산−부채) 대시보드, 부채·자산 상세 관리, 실시간 계좌 연동, 2단계 PWA 구현 상세(별도 spec).
