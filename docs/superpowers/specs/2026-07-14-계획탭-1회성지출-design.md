# 계획 탭 — 1회성 지출 설계

- 날짜: 2026-07-14
- 대상: 배추가족 PWA `📊손익` → 「계획」 탭
- 선행: `2026-07-13-가족-pnl-계획탭-design.md` (계획 탭 최초 구현)

## 1. 배경

계획 탭을 배포한 직후 재헌님이 요청했다.

> "1회성 지출 예산도 반영이 가능해? 특정 월에만 지출 이런식으로 체크해놓으면 예산에서 특정 월에만 반영되도록"

지금은 안 된다. `PlanItem`에 `endYearMonth`(언제까지)만 있고 **시작 경계가 없어서**, 모든 항목이 "지금부터 종료월까지 매달" 반영된다. 특정 월에만 태울 방법이 없다.

**용도**(재헌님 확인): ① 예상되는 큰 지출을 미리 꽂아두기 ② 이번 달만 특별히 쓸 돈.
**범위**: 특정 월 1회만. 기간(N월~M월)은 필요 없다.

## 2. 데이터 모델

`PlanItem`에 **`startYearMonth: string | null`** 하나를 추가한다. 판정 규칙이 하나로 통일된다.

> 항목은 **`시작월 ≤ 기준월 ≤ 종료월`**일 때 살아 있다. (null은 무한대로 취급)

| 항목 종류 | startYearMonth | endYearMonth |
|---|---|---|
| 매달 (월세·식비) | null | null |
| 언젠가 끝남 (자동차 할부) | null | 2027-03 |
| **1회성 (8월 산후조리비)** | **2026-08** | **2026-08** |

`oneTime` 플래그를 따로 두지 않는 이유:
- **컬럼 하나로 끝난다.** 기존 27개 항목은 전부 `startYearMonth = null`이라 **데이터 마이그레이션이 필요 없고 동작도 그대로**다.
- **판정 규칙이 두 갈래로 안 갈라진다.** `if (oneTime) ... else ...` 분기가 생기면 그 분기가 `computePlanPnl`·리스트·대조표에 각각 복제된다.
- 1회성은 `start === end`라는 **파생 조건**으로 알아낸다(`isOneTime`).

### 마이그레이션 `0019_plan_item_start_month`

```sql
alter table plan_items add column if not exists start_year_month text;
```

순수 추가(additive)라 안전하다. 기존 행은 전부 null이 되어 현행 동작이 유지된다. 순수 ASCII.

## 3. 시드 변경

### 3-1. 28번째 항목 추가 — 어머니 산후조리비

재헌님: "1회성으로 어머니 산후조리비 100만원으로 내가 예산에 지출로 잡아놨어" / "8월로 넣어줘"

| 필드 | 값 |
|---|---|
| id | `plan-once-postpartum` |
| group | `spending` (월 지출) |
| name | 어머니 산후조리비 |
| amount | 1,000,000 |
| pnlClass | `variable` (변동비) |
| conditional | false |
| startYearMonth | **2026-08** |
| endYearMonth | **2026-08** |
| sortOrder | 240 (연료비 230 다음) |

**DB에 아직 시드가 안 들어갔으므로**(`plan_items` 0행 — 재헌님이 앱을 아직 안 열었다) 시드 상수에 넣으면 최초 로드 시 28개가 함께 들어간다. 앱에서 수동 추가할 필요가 없다.

### 3-2. 예산 규칙에서 한 줄 삭제

`BUDGET_RULES`(`lib/types.ts`)에서 아래 줄을 **삭제**한다.

```
"엄마 1,000,000은 첫만남 이용권으로 충당 — 예산에서 제외",
```

이유: 그 1,000,000이 이제 **어머니 산후조리비라는 실제 1회성 지출**로 예산에 잡혔다. "예산에서 제외"는 더 이상 사실이 아니다.

남는 규칙 3줄:
```
"금액은 넉넉하게 올려 잡는다",
"현금·배찌 주식은 계획에서 잊는다",
"첫만남 이용권·안양시 200만원 등 일회성 금액은 잊는다",
```

> 세 번째 규칙("일회성 금액은 잊는다")과 이번 기능(일회성 **지출** 추가)은 충돌하지 않는다. 그 규칙이 말하는 일회성 금액은 첫만남 이용권·안양시 200만원 같은 **일회성 수입/지원금**이다. 계획에 넣지 않는 건 수입 쪽이고, 지금 넣는 건 지출 쪽이다.

## 4. 골든 값 재정의

8월에 1,000,000짜리 변동비가 생기므로 **8월 계획 P&L이 바뀐다.** 기준선(매달 반복분)과 8월을 분리해 둘 다 검증한다.

| 기준월 | 운영이익 | 의미 |
|---|---|---|
| **2026-07** (기본 화면) | **25,794** | 매달 반복분 기준선 — 손글씨 노트의 "잔액" |
| **2026-08** | **−974,206** | 산후조리비 1,000,000이 반영된 달 |
| **2026-09** | **25,794** | 기준선 복귀 |

25,794는 여전히 이 계획의 기준선이고, 앱을 열었을 때(2026-07) 히어로에 보이는 값도 그대로다.

> ⚠️ 기존 `lib/plan.test.ts`의 골든 테스트는 기준월이 `2026-08`이다. **`2026-09`로 옮겨야** 기준선 의미가 유지된다. 그리고 `2026-08` → −974,206 케이스를 새로 추가한다.

## 5. 계산 (`lib/plan.ts`)

```ts
// 시작 경계 추가 — null은 무한대
export function activeItems(items: PlanItem[], yearMonth: string): PlanItem[] {
  return items.filter(
    (i) =>
      (i.startYearMonth == null || i.startYearMonth <= yearMonth) &&
      (i.endYearMonth == null || i.endYearMonth >= yearMonth)
  );
}

// 1회성 = 시작월과 종료월이 같은 달로 못박힌 항목
export function isOneTime(i: PlanItem): boolean {
  return i.startYearMonth != null && i.startYearMonth === i.endYearMonth;
}

// 기준월 "이후"의 1회성 항목 — 「다가오는 1회성 지출」 카드용. 월 오름차순.
export function upcomingOneTime(items: PlanItem[], yearMonth: string): PlanItem[] {
  return items
    .filter((i) => isOneTime(i) && i.startYearMonth! > yearMonth)
    .sort((a, b) => a.startYearMonth!.localeCompare(b.startYearMonth!));
}
```

`computePlanPnl`·`groupTotal`·`remainingMonths`는 시그니처 변경 없음. `computePlanPnl`은 `activeItems`를 쓰므로 새 규칙을 자동으로 탄다.

## 6. 화면

### 6-1. 편집 시트 (`PlanItemForm`) — 「적용」 세그먼트

"시작월/종료월"이라는 날것을 노출하지 않는다. 세그먼트 2개로 의도를 고른다.

| 선택 | 보이는 입력 | 저장되는 값 |
|---|---|---|
| **[매달]** (기본) | "언제까지 (비우면 무기한)" | `start = null`, `end = 입력값 \|\| null` |
| **[이 달만]** | "어느 달" | `start = end = 입력값` |

- 편집으로 열 때: `isOneTime(initial)`이면 **[이 달만]**, 아니면 **[매달]**로 시작.
- **[이 달만]** 선택 시 "어느 달"이 비어 있으면 저장 불가(`canSave`에 포함).
- 세그먼트를 바꿔도 `group`·`pnlClass`는 건드리지 않는다 — 독립 축 원칙 유지.

### 6-2. 「다가오는 1회성 지출」 카드 (`UpcomingOneTime.tsx`, 신규)

위치: **계획 vs 실적 아래, 항목 리스트 위.**

- `upcomingOneTime(planItems, month)`를 월 오름차순으로 나열: `8월 · 어머니 산후조리비` + 금액
- `SectionTitle`의 `right`에 **합계**
- 행 탭 → 기존 `PlanItemForm` 편집 시트
- **`+ 1회성 지출 추가` 버튼** — 이 기능의 진입점. 「적용」이 [이 달만]으로 preselect되고, 기본 달은 **다음 달**, group=`spending`, pnlClass=`variable`
- **비어 있어도 카드를 숨기지 않는다.** 추가 버튼이 진입점이므로 항상 보여야 한다. 빈 상태엔 안내 한 줄.

**이번 달 P&L에는 안 들어간다.** `activeItems`가 걸러내므로 자동이다. 그 달로 넘어가야 잔액에 반영된다. 카드에 이 사실을 한 줄로 적는다 — 안 그러면 "왜 잔액에 안 빠지지?"로 헤맨다.

### 6-3. 리스트 배지 (`PlanItemList`)

1회성 항목은 그 달 리스트에서 **`1회성` 배지**(`Pill tone="sky"`)를 단다.

지금 로직(`remainingMonths`)대로면 `1회 남음`이 뜨는데, 이건 "할부 8회 남음"과 같은 계열로 읽혀 혼동된다. `isOneTime(i)`이면 `1회성`, 아니면 기존 `N회 남음`.

## 7. 동작 예시 (2026-07 기준, 앱 최초 오픈)

1. 히어로 잔액 **+25,794** — 종이 노트와 일치 (산후조리비는 8월이라 안 빠짐)
2. 「다가오는 1회성 지출」에 `8월 · 어머니 산후조리비 1,000,000`, 합계 1,000,000
3. 월 스위처로 **8월** → 항목 리스트 「월 지출」에 `어머니 산후조리비` + `1회성` 배지 등장, 잔액 **−974,206**
4. **9월** → 사라지고 잔액 25,794 복귀. 「다가오는 1회성 지출」은 비어 있음(8월이 지났으므로)
5. 8월로 되돌리면 다시 보인다 — **지난 1회성은 자동 삭제하지 않는다**(재헌님 결정). 과거 계획 vs 실적 기록이 보존된다.

## 8. 테스트 (`lib/plan.test.ts`)

기존 테스트에 더한다.

1. `computePlanPnl(SEED, "2026-09").operatingProfit === 25794` — **기준선 골든**(기존 테스트의 기준월을 2026-08 → 2026-09로 이동)
2. `computePlanPnl(SEED, "2026-07").operatingProfit === 25794` — 기본 화면도 기준선
3. `computePlanPnl(SEED, "2026-08").operatingProfit === -974206` — **산후조리비 골든**
4. `computePlanPnl(SEED, "2026-08").variable === 1300000 + 1000000` — 변동비로 잡힌다
5. `activeItems(SEED, "2026-08")`에 `plan-once-postpartum` 포함
6. `activeItems(SEED, "2026-07")`·`activeItems(SEED, "2026-09")`에 **미포함**
7. `isOneTime` — start===end면 true, start=null이면 false, start≠end면 false
8. `upcomingOneTime(SEED, "2026-07")` → `[plan-once-postpartum]`
9. `upcomingOneTime(SEED, "2026-08")` → `[]` (당월은 "다가오는"이 아니다)
10. `upcomingOneTime` 정렬 — 여러 개면 월 오름차순
11. `groupTotal(SEED, "spending")`은 **2,760,000 + 1,000,000 = 3,760,000** — `groupTotal`엔 기준월 인자가 없어 전체를 더한다. `Plan.tsx`가 `activeItems`로 거른 목록을 넘기므로 화면은 정상. **이 테스트는 그 사실을 못박아 둔다.**

## 9. 범위 밖 (YAGNI)

- **기간형(N월~M월)** — 데이터 모델은 지원하지만 UI로 노출하지 않는다. 재헌님이 "특정 월 1회만 필요"라고 확인했다.
- **지난 1회성 자동 삭제** — 재헌님이 "그대로 남겨둔다"를 선택했다.
- **1회성 수입** — 요청은 지출이었다. 데이터 모델상 `group=income`에 1회성을 만드는 건 가능하지만 UI 진입점(`+ 1회성 지출 추가`)은 지출용으로만 둔다.
- **`startYearMonth`로 "이번 달부터 매달" 표현** — 모델은 되지만 UI에 안 넣는다. 지금 매달 항목은 과거 달에도 나타나는데(start=null), 계획 탭은 전방 조망용이라 실해가 없다.
